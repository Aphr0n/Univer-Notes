# Типи VLAN
## VLAN за замовчуванням
VLAN за замовчуванням (Default VLAN) на комутаторі Cisco - це VLAN 1. Bсі порти комутатора належать до VLAN 1 до того моменту, поки вони явно не будуть налаштовані для участі у інших VLAN. Увесь трафік управління канального рівня пов'язаний з VLAN 1.

Слід пам'ятати про VLAN 1 такі важливі факти:
- За замовчуванням всі порти комутатора належать до VLAN 1.
- За замовчуванням VLAN з нетегованим трафіком (Native VLAN) - це VLAN 1.
- За замовчуванням управлінська VLAN (Management VLAN) - це VLAN 1.
- Не можна перейменувати або видалити VLAN 1.

Наприклад, у результатах виводу команди show vlan brief вс порти належать VLAN за замовчуванням - VLAN 1. Жодна інша VLAN явно не вказана як Native VLAN, відсутні інші активні VLAN; отже, мережа спроектована з VLAN з нетегованим трафіком тією ж, що й управлінська VLAN. Таке налаштування є загрозою безпеці мережі.

## Віртуальна локальна мережа (VLAN) 
Віртуальна локальна мережа (VLAN) — це логічне групування пристроїв у мережі, яке дозволяє їм взаємодіяти так, ніби вони знаходяться в одній локальній мережі, навіть якщо вони фізично підключені до різних комутаторів.

## Native VLAN
**Native VLAN** — це спеціальна VLAN на транковому порту 802.1Q, у яку поміщається **нетегований трафік**.
Тобто, якщо кадр приходить на транковий порт **без тегу VLAN**, комутатор автоматично відносить його до Native VLAN. І навпаки — трафік, що виходить з Native VLAN через транковий порт, надсилається **без тегу**.

За замовчуванням на комутаторах Cisco Native VLAN — це **VLAN 1**.

Нетегований трафік може з'являтися у двох випадках:

- генерується самим комутатором (службовий трафік)
- надходить від застарілих пристроїв, які не підтримують 802.1Q

Оскільки VLAN 1 є дефолтною і широко відомою, залишати її як Native VLAN вважається **небезпечним** (є ризик атак типу VLAN hopping). Тому рекомендується призначити Native VLAN якусь невикористовувану VLAN (наприклад, VLAN 999) на всіх транкових портах усіх комутаторів домену.

## Управлінська VLAN

**Управлінська VLAN (Management VLAN)** — це виділена VLAN, через яку адміністратор отримує **віддалений доступ до керування комутатором**.

Саме через неї передається трафік протоколів адміністрування:
- **SSH / Telnet** — віддалений термінальний доступ
- **HTTPS / HTTP** — веб-інтерфейс керування
- **SNMP** — моніторинг мережі

За замовчуванням на Cisco це **VLAN 1**, але з міркувань безпеки рекомендується створити окрему VLAN спеціально для управління і відділити її від звичайного користувацького трафіку. Так, якщо зловмисник потрапить до користувацької VLAN, він не матиме прямого доступу до управлінського інтерфейсу комутатора.

## **Голосова VLAN (Voice VLAN)** — це окрема VLAN, виділена спеціально для передавання VoIP-трафіку.

Потреба в окремій VLAN пояснюється особливими вимогами голосового трафіку — він дуже чутливий до затримок і втрат пакетів, тому його потрібно ізолювати від звичайного трафіку даних і надати йому вищий пріоритет.

**Типове підключення** виглядає так:

```
PC → IP-телефон → комутатор
```

IP-телефон виступає міні-комутатором: він має два порти і підключається до мережі сам, а до нього підключається комп'ютер. При цьому порт комутатора одночасно обслуговує **дві VLAN**:

- **Голосова VLAN** (наприклад, VLAN 150) — для трафіку телефону, **тегований**
- **VLAN даних** (наприклад, VLAN 20) — для трафіку ПК, **нетегований**

Саме тому такий порт налаштовується як **access-порт з додатковою voice VLAN**, а не як звичайний транк. На Cisco це виглядає так:
```CISCO
switchport mode access
switchport access vlan 20
switchport voice vlan 150
```

## Транкові канали VLAN 

### Що таке транк і навіщо він потрібен?

Щоб пристрої однієї VLAN на різних комутаторах могли спілкуватися, потрібен **транковий канал** — один фізичний кабель між комутаторами, по якому йде трафік **усіх VLAN одночасно**.
Без транку довелося б тягнути окремий кабель для кожної VLAN — це нераціонально.

### Як комутатор розрізняє, якій VLAN належить кадр Тегування 802.1Q

Стандартний Ethernet-кадр не містить інформації про VLAN. Тому коли кадр іде через транк, комутатор **вставляє в нього 4-байтний тег** (IEEE 802.1Q). Тег містить:

|Поле|Розмір|Зміст|
|---|---|---|
|Type (TPID)|2 байти|Завжди `0x8100` — сигнал що кадр тегований|
|PRI|3 біти|Пріоритет (для QoS)|
|CFI|1 біт|Сумісність з Token Ring|
|**VID**|**12 біт**|**Номер VLAN (1–4094)**|

Після вставки тегу комутатор перераховує **FCS** (контрольну суму), бо вміст кадру змінився.
На виході з транку — тег знімається, і кінцевий пристрій отримує звичайний кадр.

### Native VLAN — особливий випадок

**Native VLAN** — це єдина VLAN, трафік якої через транк іде **без тегу**. За замовчуванням це VLAN 1.
Важливі правила:
- Нетегований кадр на транковому порту → автоматично потрапляє в Native VLAN
- Якщо прийшов тегований кадр з ID рівним Native VLAN → комутатор Cisco **відкидає** його (це помилка конфігурації)
- На обох кінцях транку **Native VLAN повинна збігатися**, інакше буде проблема безпеки (VLAN hopping)

### Голосова VLAN (Voice VLAN)
IP-телефон Cisco — це по суті **маленький 3-портовий комутатор**:
- Один порт іде до комутатора мережі
- Один — внутрішній (для самого телефону)
- Один — для підключення ПК
Тому один фізичний порт комутатора обслуговує **дві VLAN**:
- **Голосова VLAN** (наприклад, VLAN 150) — для дзвінків, з пріоритетом QoS
- **Data VLAN** (наприклад, VLAN 20) — для трафіку ПК
Це імітує міні-транк між телефоном і комутатором.

### Коротко — схема роботи

```
ПК (VLAN 10) → комутатор додає тег [VID=10] → іде по транку
                                               → інший комутатор знімає тег → ПК (VLAN 10)
```

### Діапазони VLAN

На комутаторах Cisco є два діапазони:

**Стандартний (1–1005)**

- VLAN 1 — стандартна, створюється автоматично, видалити не можна
- 1002–1005 — зарезервовані для старих технологій (Token Ring, FDDI)
- Конфіг зберігається у файлі `vlan.dat` у флеш-пам'яті

**Розширений (1006–4094)**

- Для великих мереж/провайдерів
- Зберігається у running-config
- Потребує VTP transparent mode

Ліміт 4094 пояснюється тим, що поле VLAN ID у заголовку 802.1Q — лише **12 біт** (2¹² = 4096, але 0 і 4095 зарезервовані).

### Основні команди

**Створення VLAN:**

```
S1(config)# vlan 20
S1(config-vlan)# name student
```

**Призначення порту до VLAN:**

```
S1(config)# interface fa0/6
S1(config-if)# switchport mode access
S1(config-if)# switchport access vlan 20
```

**Повернення порту до VLAN 1:**

```
S1(config-if)# no switchport access vlan
```

---

### VLAN даних + голосова VLAN

Один порт може одночасно обслуговувати два типи трафіку — якщо до нього підключений IP-телефон з ПК за ним:

```
S3(config-if)# switchport access vlan 20      ← дані (ПК)
S3(config-if)# switchport voice vlan 150      ← голос (IP-телефон)
S3(config-if)# mls qos trust cos              ← QoS для голосу
```

# DTP — Протокол динамічного транкування

> [!info] Що таке DTP? **DTP (Dynamic Trunking Protocol)** — фірмовий протокол Cisco для автоматичного узгодження транкових каналів між двома комутаторами. Увімкнений **за замовчуванням** на Catalyst 2960/3650.

---

## Режими роботи порту

|Режим|Поведінка|
|---|---|
|`access`|Завжди access, **ніколи** не стане транком|
|`dynamic auto`|**Пасивний** — стає транком тільки якщо сусід ініціює (`trunk` або `desirable`). Режим за замовчуванням|
|`dynamic desirable`|**Активний** — сам намагається сформувати транк|
|`trunk`|Завжди транк, незалежно від режиму сусіда|

---
## Таблиця узгодження

|        | auto      | desirable | trunk     | access     |
|--------|-----------|------------|-----------|------------|
| **auto** | ❌ access | ✅ trunk   | ✅ trunk  | ❌ access |
| **desirable** | ✅ trunk | ✅ trunk | ✅ trunk | ❌ access |
| **trunk** | ✅ trunk | ✅ trunk | ✅ trunk | ⚠️ обмежено |
| **access** | ❌ access | ❌ access | ⚠️ обмежено | ❌ access |

**Пастка:**  
Два порти в режимі `auto` з обох боків — транку не буде, бо обидва пасивні і жоден не ініціює.
## Команди

### Статичний транк без DTP (рекомендовано)

```ios
switchport mode trunk
switchport nonegotiate
```

> Використовувати для не-Cisco пристроїв або коли потрібна стабільна конфігурація без переговорів.

### Повернути DTP (пасивний режим)

```ios
switchport mode dynamic auto
```

### Перевірити стан DTP на порту

```ios
show dtp interface fa0/1
```

---

## Найкращі практики

> [!tip] Правила гарного тона
> 
> - **Потрібен транк** → `trunk` + `nonegotiate` — явно, без автоматичних переговорів
> - **Транк не потрібен** → `access` — повністю вимикає DTP на порту
> - **Підключення до не-Cisco пристроїв** → завжди `nonegotiate`, інакше можливі некоректні конфігурації

> [!danger] Безпека DTP може бути вектором атаки типу **VLAN hopping** — зловмисник може надіслати DTP-кадри та спровокувати утворення небажаного транку. Тому на всіх портах доступу (кінцеві пристрої, не-Cisco) слід явно вказувати `access`.
