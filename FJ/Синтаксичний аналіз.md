Головним завданням синтаксичного аналізу є перевірка, чи послідовність токенів, отриманих за допомогою лексичного аналізу, відповідає граматичним правилам мови програмування.
- працює з токенами, а не з символами,
- дотримується структури, визначеної в CFG/EBNF,
- виявляє синтаксичні помилки,
- готує вхідні дані для подальших фаз перекладу

# Вихідні дані парсера: дерево виводу проти AST

| Дерево виведення                              | AST                                                         |
| --------------------------------------------- | ----------------------------------------------------------- |
| фіксує весь процес виведення,                 | видаляє синтаксичний баласт,                                |
| містить усі термінали та нетермінали,         | фіксує основну структуру програми,                          |
| безпосередньо пов'язаний з певною граматикою. | більше підходить для семантичного аналізу та інтерпретації. |
`Дерево виведення` $\xrightarrow{трансформація}$ `AST`

# Два основні принципи побудови дерева

| Top-Down                                            | Bottom-Up                           |
| --------------------------------------------------- | ----------------------------------- |
| починається з початкового символу,                  | починає з вхідних токенів,<br>      |
| розширює нетермінали,                               | виконує скорочення,                 |
| іде від кореня до листя.                            | працює від листя до кореня.<br>     |
| пряме посилання на граматику,<br>                   | охоплює ширший клас мов,<br>        |
| придатний для підходів навчання на основі навчання, | зазвичай механізм shift-reduce,<br> |
| легший для пояснення,                               | потужніший, але складніший,         |
| чутливий до граматичної структури.                  | придатний для LR-парсерів.          |

# Вибір правила та повернення
- Нетермінал може мати кілька альтернатив, тому синтаксичний аналізатор повинен вибрати правило.
- Наївним рішенням є спроба альтернатив методом перебору та відмова від попереднього методу, якщо це не вдається.
- Такий підхід неефективний і може не завершитися через невідповідну граматику.

# Рекурсивний спуск: основна ідея
- Ми призначаємо процедуру або функцію кожному нетерміналу.
-  Тіло процедури копіює праву частину правила.
-  Виклики процедур дотримуються структури виведення.

## Приклади застосування
- Прогнозний рекурсивний спуск вимагає граматики, модифікованої до форми LL(1) або, загалом, LL(k).
- Одного символу (або k символів) попереднього пошуку має бути достатньо для вибору правильної альтернативи.
- Тому граматика не повинна містити конфліктів, які синтаксичний аналізатор не може вирішити локально.

## BNF, розгалуження та множина FIRST

### Елементи EBNF та їх реалізація

| Елемент EBNF/BNF | Реалізація                       |                                         |
| ---------------- | -------------------------------- | --------------------------------------- |
| A ::= ...        | `def A(): ...`                   |                                         |
| нетермінал A     | виклик `A()`                     |                                         |
| термінал a       | `consume(a)`                     |                                         |
| α                | β                                | розгалуження за `FIRST(α)` і `FIRST(β)` |
| [ α ]            | `if current in FIRST(α): ...`    |                                         |
| { α }            | `while current in FIRST(α): ...` |                                         |

## Множина FIRST

**FIRST(α)** — це множина термінальних символів, якими може починатися рядок, виведений із α.

Ця множина використовується як **точка прийняття рішення парсером**:
- визначає, яку альтернативу вибрати
- керує входом у розгалуження або цикл

## Правила проєктування рекурсивного парсера

1. Для кожного правила створюється процедура з назвою нетермінала.
2. Нетермінал у правій частині → виклик відповідної процедури.
3. Термінал → перевірка і споживання токена (`consume`).
4. Порядок символів у коді відповідає граматиці.
5. Альтернативи та необов’язкові конструкції реалізуються через умовні оператори.
6. Ітерації реалізуються через цикл `while`.
7. Запуск парсера здійснюється через стартовий символ.

## Потік керування як дерево виведення

Послідовність викликів процедур не є випадковою — вона відповідає:
- дереву виведення (derivation tree)

Парсер:
- перевіряє синтаксичну коректність
- відтворює структуру виразу
