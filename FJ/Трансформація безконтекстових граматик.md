Трансформація граматики — це процес створення іншої граматики з заданої граматики зазвичай еквівалентної з точки зору згенерованої мови, але зручнішої для аналізу, реалізації або подальшої модифікації.

>[!info] Ключова ідея
>Ми не хочемо змінювати рядки, які містить мова. Ми хочемо змінити зручність роботи з заданою граматикою.

Типові причини трансформації:
- видалення непродуктивних або недоступних символів,
- видалення небажаних типів правил,
- адаптація граматики до певного типу парсера.

# Основні трансформаційні методи

Найважливіші перетворення, з якими будемо працювати:
1. видалення надлишкових символів,
2. видалення ε-правил,
3. видалення простих правил,
4. видалення лівої рекурсії,
5. ліва факторизація.

---

## Термінологія

### Редукована граматика
Граматика не містить надлишкових символів.

### Власна граматика
Граматика не містить:
- надлишкових символів,
- циклів,
- ε-правил  
  (окрім стандартного винятку для початкового символу).

---

# Надлишкові символи: основні поняття

## Недосяжний символ
Символ, який неможливо вивести з початкового символу.

## Надлишковий символ
Символ є:
- або недосяжним,
- або хоча й з’являється у деякій сентенціальній формі, але ніколи не приводить до термінального слова.

> [!info] Кожен недосяжний символ є надлишковим, але не кожен надлишковий символ є недосяжним.

## Під час видалення надлишкових символів розглядаємо два питання:

1. Які нетермінали можуть генерувати термінальні рядки?
2. Які символи після очищення граматики взагалі досяжні зі стартового символу?

---

# Процедура видалення надлишкових символів

1. Будуємо множину `NT` нетерміналів, які можуть генерувати термінальні рядки.
2. Видаляємо правила, що містять нетермінали поза `NT`.
3. Будуємо множину досяжних символів `VD`.
4. Видаляємо символи й правила поза `VD`.

Досяжність має сенс перевіряти лише після видалення непродуктивних частин.
Інакше до множини досяжних символів потрапили б також символи, що ведуть у «глухі гілки».

# Побудова множини `NT`

```text
NT ← ∅
```

Повторно додаємо кожен нетермінал `A`, для якого існує правило:

```text
A → α
```

де права частина `α` складається лише:
- з терміналів,
- або з уже відомих елементів множини `NT`.
Якщо я можу згенерувати термінальні рядки зі символів правої частини,
то можу згенерувати їх і з лівої частини правила.

Алгоритм завершується в момент, коли множина `NT` у наступній ітерації більше не змінюється.


# ε-правила

Правило вигляду `A → ε` називається **ε-правилом**.

## Призначення ε-правил

ε-правила дозволяють нетерміналу «зникнути», що природно при описі необов’язкових частин мови.

Водночас вони ускладнюють:
- алгоритми аналізу,
- інші трансформації граматик.

## Виняток

Правило `S → ε` може залишитися, якщо початковий символ `S` не зустрічається у правій частині жодного іншого правила(по суті стартувальний символ).

---

# Процедура видалення ε-правил

1. Будуємо множину `Nε` нетерміналів, які можуть вивести `ε`.
2. Модифікуючи правила, створюємо всі релевантні комбінації з пропуском символів із `Nε`.
3. Видаляємо початкові ε-правила, окрім коректно обробленого випадку стартового символу.

---

## Суть проблеми

Якщо у правій частині міститься кілька символів із `Nε`,
потрібно врахувати всі можливі комбінації:
- присутності,
- та пропуску цих символів.

---

# Побудова множини `Nε`

Починаємо з множини

```text
Nε = ∅
```

## Алгоритм

1. Повторно додаємо до `Nε` кожен нетермінал `A`, для якого існує правило `A → α` де `α` складається лише із символів, які вже належать до `Nε`.
2. Одразу додаємо нетермінали з правилом `A → ε`
3. Потім додаємо інші нетермінали, які можуть зводитися до `ε` непрямим шляхом.


# Прості правила та цикли

Просте правило має вигляд `A → B`, де `A` і `B` — нетермінали.

## Властивості простих правил

Такі правила самі по собі нічого «не породжують». Вони лише перенаправляють деривацію:
- з одного нетермінала
- на інший.

## Цикл

Цикл означає існування виведення `A ⇒⁺ A`

> [!info] Важливе зауваження 
>Якщо граматика не містить простих правил, і не містить ε-правил, то вона не має циклів. Але обернене твердження неправильне. Відсутність циклів не гарантує, що в граматиці немає простих правил, або ε-правил.

---

# Процедура видалення простих правил

Для кожного нетермінала `A` будуємо множину `N_A` усіх нетерміналів, які досяжні з `A` лише за допомогою простих правил.

## Далі виконуємо:

1. Видаляємо всі прості правила.
2. Для кожного `B ∈ N_A` переносимо в `A` всі непрості правила нетермінала `B`.

Тобто замість ланцюжка `A ⇒ B ⇒ α` додаємо безпосередньо правило `A → α`

# Ліва рекурсія

Нетермінал `A` є ліворекурсивним, якщо існує виведення `A ⇒⁺ Aβ`. Під час рекурсивного спуску процедура може викликати саму себе ще до обробки хоча б одного символу вхідного рядка і як результат виникне  нескінченна рекурсія.

## Види лівої рекурсії

### Прямa ліва рекурсія

Наприклад:

```text
A → Aα
```

### Непряма ліва рекурсія

Наприклад:

```text
A → Bα
B ⇒⁺ Aβ
```


# Стандартна схема усунення прямої лівої рекурсії

Якщо маємо правила

```text
A → Aα₁ | ··· | Aαₙ | β₁ | ··· | βₘ
```

де жодне `βᵢ` не починається символом `A`,
замінюємо їх на:

```text
A  → β₁A′ | ··· | βₘA′ | β₁ | ··· | βₘ
A′ → α₁A′ | ··· | αₙA′ | α₁ | ··· | αₙ
```

Іншими словами, рекурсію переносимо до нового допоміжного нетермінала так, щоб вона запускалась лише після обробки «не ліворекурсивного» початку.

# Ліва факторизація

Якщо два правила починаються однаковим префіксом, наприклад:

```text
A → αXβ₁
A → αYβ₂
```

то виділяємо спільний префікс:

```text
A → αB
B → Xβ₁ | Yβ₂
```

Парсер спочатку приймає рішення за спільним початком. Лише потім визначає, яке продовження потрібно використати.

---

# Ліва факторизація арифметичної граматики

Із трансформованої граматики можна отримати компактнішу форму:

```text
E  → TE′′
E′′ → +TE′′ | ε

T  → FT′′
T′′ → ∗FT′′ | ε

F  → num | (E)
```

Такий запис з точки зору реалізації дуже близький до практичного запису в EBNF.

---

# Чому ліва рекурсія не працює при рекурсивному спуску

Якщо наївно реалізувати правило

```text
E ::= E "+" T | T
```

отримаємо схему:

```python
def E():
    E()
    if current_symbol == '+':
        get_next_symbol()
        T()
```

> [!warning] Проблема в тому що виклик `E()` відбувається одразу повторно, без просування по вхідному рядку. Парсер ніколи не доходить до термінала, і як наслідок рекурсія не завершується.

---

# Практична реалізація лівоасоціативних операторів в EBNF

Для лівоасоціативного оператора природною є реалізація через ітерацію:

```python
def E():
    T()
    while current_symbol == '+':
        get_next_symbol()
        T()
```

## Пояснення

- синтаксично це повторення однакової конструкції,
- семантично зберігається ліва асоціативність,
- реалізаційно уникаємо лівої рекурсії.

---

# Повністю рекурсивна реалізація після трансформації

Після трансформації граматики парсер можна записати рекурсивно:

```python
def E():
    T()
    E1()

def E1():
    if current_symbol == '+':
        get_next_symbol()
        T()
        E1()
```

Рекурсія переноситься до допоміжного нетермінала і запускається лише після обробки
релевантного префікса вхідного рядка.
