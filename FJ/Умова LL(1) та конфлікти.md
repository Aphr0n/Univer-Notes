Граматика є **LL(1)**, якщо для кожної пари різних продукцій

```text
A → α
A → β
```

виконується:
```text
FIRST(α) ∩ FIRST(β) = ∅
```

---

## Умова для ε-правил

Якщо: `text ε ∈ FIRST(α)` то додатково має виконуватись: `text FIRST(β) ∩ FOLLOW(A) = ∅`

# Найчастіші конфлікти

## FIRST/FIRST конфлікт

Виникає, коли дві продукції можуть починатися однаковими терміналами
```text
FIRST(α) ∩ FIRST(β) ≠ ∅
```

- одна продукція може бути ε
- і її FIRST перетинається з FOLLOW нетермінала


## Загальний критерій

Якщо в одній клітинці таблиці з’являється більше одного правила,
граматика **не є LL(1)**.


# LL(1) таблиця та правила заповнення

Таблиця:

```text
M[A, a]
```

де:
- рядки → нетермінали
- стовпці → вхідні символи (включно з `$`)

## Для кожної продукції A → α обчислюємо PREDICT

### 1. Якщо a ∈ FIRST(α) \ {ε}

```text
M[A, a] = A → α
```

### 2. Якщо ε ∈ FIRST(α)

тоді для кожного`text b ∈ FOLLOW(A)` вписуємо `text M[A, b] = A → α`

---

## Підсумок PREDICT

```text
PREDICT(A → α) =
(FIRST(α) \ {ε}) ∪ FOLLOW(A)   (якщо ε ∈ FIRST(α))
```

## Властивість таблиці

- порожня клітинка → помилка  
- більше ніж 1 правило → конфлікт  
- рівно 1 правило → коректно  

# Граматика виразів для LL(1)

```text
E  → T E′
E′ → +T E′ | ε

T  → F T′
T′ → *F T′ | ε

F  → (E) | id
```

## Властивості

- немає лівої рекурсії  
- добре структурована  
- зберігає пріоритет `*` над `+`  


# LL(1) таблиця для граматики виразів

|     | id       | +          | *          | (        | )      | $      |
| --- | -------- | ---------- | ---------- | -------- | ------ | ------ |
| E   | E → T E′ |            |            | E → T E′ |        |        |
| E′  |          | E′ → +T E′ |            |          | E′ → ε | E′ → ε |
| T   | T → F T′ |            |            | T → F T′ |        |        |
| T′  |          | T′ → ε     | T′ → *F T′ |          | T′ → ε | T′ → ε |
| F   | F → id   |            |            | F → (E)  |        |        |

---

## Висновок

У кожній клітинці таблиці
є не більше одного правила,
тому граматика є LL(1).

---

# Алгоритм LL(1) парсера (табличний)

## Використовується:

- вхід, завершений символом `$`
- стек, ініціалізований як `S$`

## Кроки:

1. якщо на вершині термінал і він збігається з входом → знімаємо обидва  
2. якщо на вершині нетермінал `A` → беремо правило з `M[A, a]`  
3. якщо правила немає → помилка  
4. якщо залишились лише `$` і `$` → прийняття  

# Рекурсивний спуск

Табличний LL-парсер має прямий процедурний еквівалент.

## Основна ідея

- для кожного нетермінала створюється окрема процедура  
- вибір продукції відбувається за поточним токеном  
- виклики процедур повторюють структуру виведення  

як результат парсер стає дуже зрозумілим і структурованим.

## Приклад

Нетерміналам:

```text
E, T, F
```

відповідають процедури:

```text
parseE()
parseT()
parseF()
```

---

# Переваги та обмеження LL-аналізу

## Переваги

- концептуально простий підхід  
- легка реалізація  
- добра читабельність коду парсера  
- природний зв’язок із безконтекстними граматиками  
- зручний для навчання основ синтаксичного аналізу  

---

## Обмеження

- не кожна безконтекстна граматика є LL(1)  
- граматику часто потрібно трансформувати  
- для складніших мов використовують сильніші методи (наприклад LR-парсери)  

---

# Зв’язок стеку, LL-парсера та компілятора

## Стековий автомат

Це загальна модель для безконтекстних мов.

## LL-парсер

Конкретний детермінований механізм синтаксичного аналізу.

## Використання стека

Навіть у LL-аналізі використовується стек,
але він представляє:
- очікувані символи парсера

---

## Конвеєр компіляції

```text
джерельний код → токени → LL-парсер → AST → подальші етапи
```

## Роль парсера

Парсер є «містком» між:
- лінійним потоком токенів  
- деревною структурою програми  


---
# Синтаксичні помилки

Парсер може виявляти:

- відсутній операнд  
- відсутню закривну дужку  
- зайвий оператор  
- неправильний порядок конструкцій  

## Приклад помилки

```text
id + * id
```

Такий рядок не може бути виведений
згідно з граматикою.

## Практичний аспект

У реальних компіляторах важлива не лише детекція помилки,
але й її зрозуміла діагностика.
